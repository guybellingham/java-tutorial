The package statement.

 protecting classes

You should use the statement package package-name; at the top of your source file. It must be the first statement, and is used to group classes that share a common purpose. The package-name often has several parts, separated by a full-stop. Here are some packages that come with Java:

  • java.lang - is a special package that is automatically made available to any Java class you code. You don't have to import this package. It contains the foundations of the language itself, with classes like String, Object and System (see also java.lang.ref and java.lang.reflect).
  • java.util - this package contains 'utility' classes like Date, Calendar and the famous Java collection classes and interfaces.
  • java.io - contains classes for reading and writing byte and character streams, including InputStream, OutputStream, FileReader and FileWriter.
  • java.net - contains classes that help you program with network protocols, such as Socket and URL.
  • java.applet - this package contains the generic Applet class, which you can extend to build your own Applets.

Packages allow you to organize and 'share' your classes. With a statement of the form import package-name.classname; you can make a class in another package, available to the class you are coding. You don't have to do this for other classes that are in the same package as the one you are coding, they are automatically available to the Java compiler.
If you replace the classname with an asterisk, it acts as a 'wildcard' and makes all the public classes in the other package available for use.
An import, statement saves you from having to spell-out the fully qualified name of a class (which is package-name.Class-name), that is required by the Java compiler to unambiguously and uniquely identify that class.
So for example the Shape class shown below uses an import statement to reference the java.awt.Rectangle class. 

package my.shapes;
import java.awt.*; 
abstract class Shape { 
  // A Shape has a bounding box
  Rectangle box;
  // ..etc 
}

Note: The Java compiler actually only imports the classes it needs, an import with an asterisk does not lead to large executable (.class) files.
Gotcha! Now there is a potential problem with the above example. If I define my own class called Rectangle, and put it in the same package as the Shape class, I will get an error when compiling the Shape class, saying that the definition of the variable box is using an ambiguous class called Rectangle ? The compiler doesn't know which Rectangle class to use, because it has 2 to choose from, java.awt.Rectangle and my.shapes.Rectangle.
In cases of ambiguity I need to use the fully qualified class name shown below. 

  // A Shape has a bounding box
  java.awt.Rectangle box;
  • Packages allow you to organize classes into libraries and control access to their member methods and variables. On the following page, we will discuss object encapsulation, and we will find out that classes within the same package, have access to each others variables and methods (unless they are defined with the private keyword). Access between classes in different packages is much more restricted.
  • The package name tells Java where to find the executable file at runtime. Package names actually represent the 'path' (file folder structure) to the executable .class file, that Java loads at runtime. For example, if you define the Shape and Circle classes with the first statement package my.shapes; At runtime the JVM will expect the Shape class file to be in this location my/shapes/Shape.class (relative to the current working directory which is also known as the 'codebase'). If you do not explicitly assign a package name to a class, it will be assigned to the 'default' package, which represents the current working directory for your Java program or applet.
