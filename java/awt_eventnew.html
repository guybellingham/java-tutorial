java.awt.AWTEvent

 Event Listeners

The 'old' Java 1.0 event propagation and handling mechanism has some drawbacks: Because you cannot pick which events are sent to your component and its containers, your application is often flooded with events that it does not want to handle. The performance was slow (especially if your application handled mouse movements). There was no support for more sophisticated user interactions, such as mouseless operation (tabbing and shortcut keys), ability to cut & paste and drag & drop. The old event handling strategy forced you to either subclass every component to add useful event handling methods or sort through all the events (propagated up through the containment heirarchy) in a top level container.

The Java 1.1 event handling, represents the different kinds of events as seperate classes (found in the java.awt.event package) and is based upon the concepts of an event 'source' and an event 'listener'. When a user interacts with a component on screen, the AWT notifies all the objects (if any) that have registered an interest in that type of event with the component. You register as a listener by calling one of the addXXXListener( ) methods, where XXX represents the type of events you need to be notified about. For example

//Register object ml as a listener for mouse activity on myButton
myButton.addMouseListener(ml);

A Listener is a class that implements one of the EventListener interfaces.

The event objects that your listeners receive are tailored to the type of event they represent. For example a MouseEvent has getX( ) and getY( ) methods that return the coordinates to the event, whereas a KeyEvent has a getKeyCode( ) method to return an integer that can be used to identify the key hit (using VK_ constants defined in KeyEvent).

The specific type of user interface event is now represented by a subclass of AWTEvent.

Note: The 'old' event model is still supported, in Java 1.x and 2.0, however you cannot mix both event models in the same class file. Only the more recent web browsers contain a JVM that supports the 'new' event handling scheme. The new SWING user interface only supports the newer event model.

figure: AWTEvent has generic properties that all these new event types share, such as an integer id and the method getId( ).

The first four Event types listed below are termed 'low level' events and represent raw user input. The last five events are called 'semantic' events and are higher level or compound user interactions such as a button click (mouse pressed + released), or a scroll bar adjustment (mouse drag and release). This is just terminology since Java treats all these events in the same way.

ComponentEvent A component has been moved, resized, shown or hidden.
FocusEvent A component has received or lost the input focus.
MouseEvent The mouse pointer has entered or left the screen space, the mouse button has been pressed, released, clicked on the component, the mouse has moved or been dragged over the component.
KeyEvent A key or combination of keys input to the component.
ItemEvent A new item has been selected / deselected in a List, Checkbox, CheckboxMenuItem or Choice (drop down) component.
WindowEvent The window has opened / closed, been activated / deactivated or has been iconified / restored.
ContainerEvent A component has been added or removed from the container (which is itself a component).
ActionEvent A component has been 'activated' in some way e.g a Button has been clicked, a string has been entered into a TextField.
AdjustmentEvent A scrollbar has been dragged.
