Lifecycle of a thread

 Thread methods

During its lifetime, a thread spends some time executing and some time in any of several non-executing states. The scheduling of threads is an operating system dependent function, Java does not guarantee a particular behavior. Java, does provide for some control of thread execution : the setting of thread priorities, and synchronization and communication between threads.

A thread is 'alive' between the time it is started and when it stops.
Once started, a thread becomes 'ready' to run.
The operating system scheduler, decides which thread gets control of the CPU.
A thread automatically stops once its' run( ) method exits.
A thread can be forced to stop, by calling the stop( ) method on the thread object.

diagram: of the various 'states' a thread of execution may enter during its lifetime.

Many methods that perform some sort of input or output, have to wait for something to happen (locating of a file or opening of a network connection) in the outside world before they can proceed. A thread that has to wait on the slow moving outside world, must reliquish control of the CPU, so that useful work can continue and the computer remains responsive.
All this is handled by the runtime environment, and threads waiting for resources are said to be 'blocked'.

'Monitor states' represents threads that voluntarily issue a wait( ) call upon a Java object of some kind. The methods wait( ) and notify( ) are implemented in the java.lang.Object class and cannot be overidden by your classes. Note: The methods you can use to influence a threads' execution are examined on the next page.

Modern operating systems use 2 general models for multi-tasking.

Preemptive - The system scheduler picks the highest priority thread , that is ready to run, from amongst competing Threads. If a higher priority thread of work becomes ready to run, it preempts a lower priority one and takes over the CPU.
This scheme works well for cooperating threads, that are working on related tasks. However a high priority thread can monopolize the CPU.
Time Sliced - This model builds upon the notion of a competing pool of threads, with differing priorities, but the scheduler occasionally interrupts the executing thread, giving other ready threads the chance to execute at regular intervals.
This scheme works well for multiple independent threads of work.
