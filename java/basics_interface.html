Defining an interface.

 compile and run examples

An interface definition serves as a 'protocol' of behavior, which objects can share. An interface is a way of ‘laying out’ the variables and methods of a class, but not actually implementing the code that gives them ‘life’ when they are executed. You define an interface, in a manner similar to a class that defines only abstract methods, and constant class variables.
For example here is a simple interface for Shape objects that want to be able to rotate about their center points, in a standard manner : 

public interface Rotatable {
  /* variables are implicitly public, static and final */
  public int CLOCKWISE = 1;
  public int ANTICLOCKWISE = 0;
  /* an abstract method with no { body of code } */
  public void rotate(float angle, int direction);
}

Notice that you do not use the abstract keyword on the methods and final static keywords on the variables, since these are the automatic defaults for an interface.

An interface can be implemented by any class without forcing an inheritance relationship. In fact Java allows classes to implement as many interfaces as they choose without having to extend them. A class does not ‘inherit’ any behaviors from an interface; instead a class implements an interface, by providing appropriate behavior for the methods described in that interface.
A Rectangle implements the standard Rotatable interface like this : 

public class Rectangle extends Shape implements Rotatable {
  /* member variables and methods here */
  
  /* Rectangle must provide an implementation for the rotate( ) method */
  public void rotate(float angle, int direction){
    //DO rotation calculations here
  }
}

An interface is similar to a class, in that it defines a new object type. An interface can be used in a variable definition and passed as a parameter to a method. Any class that implements an interface is considered to be an instanceof that type of interface. The previous discussion of Polymorphism, applies equally to class and interface definitions.

Interfaces are needed, because Java was designed to allow only single inheritance.
A class of object in Java can have only 1 parent (super) class. This avoids some of the ambiguities of multiple inheritance that plague languages like C++. However, some designs are best served when an object can ‘behave like’ more than one other type of object. Java keeps the inheritance hierarchy simple, but has a separate and more flexible mechanism (interfaces) to provide for those classes of object that need to implement the properties and behaviors of more than one other class of object.

Interfaces can extend other interfaces. Just like classes, you can define a new type of interface, by extending (inheriting from) an existing interface. Unlike classes, interfaces can inherit from more than one other interface type. So rather confusingly, multiple inheritance is permitted when defining interfaces. This is because, interfaces do not define any implementation, so ambiguous situations, like inheriting the same method (the behavior) from two different parent classes, cannot occur.

Warning! It makes no sense to use the private or protected access modifiers within an interface. The whole point of an interface, is to layout public method signatures. Similarly you cannot specify synchronized or volatile members within an interface, since it does not define an implementation.

Warning! Once you have defined an interface, you cannot change the methods or add new ones, without changing every class that implements that interface ! Remember an interface is an agreed upon 'protocol' which classes must implement fully.
