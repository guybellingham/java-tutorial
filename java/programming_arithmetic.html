Java Arithmetic.

 assigning and comparing

The Java arithmetic operators are used to combine or compare numeric data variables (as well as numeric literals). Some operators can have a single argument, like the unary plus or negative sign (+x or -y), and others (binary operators) have 2 operands like addition (x + y) or multiplication (x * y).

Here are some general arithmetic rules to bear in mind :

  • Evaluation of an expression proceeds from left to right.
    But assignment operations (a = b = c) are evaluated right to left.
  • If evaluation of an expression cannot complete, an ArithmeticException is abruptly thrown, and no part of the expression to the right of the point of failure is evaluated.
  • The order of evaluation respects parentheses and the natural precedence of operators.
  • The operands of an expression are fully evaluated before the operation between them is performed.

Java automatically performs any necessary type conversions on operands before evaluating an expression. This usually involves a widening of operands as shown in the table below.
Beware of combining a long and float in the same expression, since as noted on the datatypes page a float may only approximate the precision of a long.

Type of operands Result 
IF neither operand is a float, double or long datatype. integer arithmetic is used. Notice that byte, short and char are widened to an int. int
IF neither operand is a float or a double and at least one operand is a long, 64-bit integer arithmetic is performed. long
IF at least one operand is a float and neither operand is a double, 32-bit floating point arithmetic is performed. float
IF at least one operand is a double, then double precision floating point arithmetic is performed, and the result is a double. double

Casting is jargon for converting between types. The Java compiler can cast numeric datatypes to a wider datatype automatically when needed. This is often termed promotion and preserves the signed value (by extending the sign bit for integral types). Casting to a narrower datatype (one with lesser precision) must be done explicitly (by prefixing with the new datatype in parentheses) because it discards the high order bits! Casting from a float or double to an integral type discards the fractional part of the value. Note that you cannot cast between the boolean datatype and any of the other datatypes. Also you cannot cast a fundamental datatype to a class or interface type (and vice versa).

Gotcha! You can get compile time errors when assigning the result of an arithmetic expression to a byte or short data variable. The Java runtime doesn't handle anything smaller than 32-bits when doing arithmetic, and as noted above requires an explicit cast, when narrowing the precision of an arithmetic result or datatype.

byte b1 = 127; //OK
byte b2 = 0x80; //compile error type mismatch - byte cant hold +128
byte b3 = b1 / 2; //compile error type mismatch - result is an int

Gotcha! Do not confuse the assignment operator (single equals =), with the double equals ==, which is used for comparing the equality of 2 expressions.

Using parentheses ( and ), you should specify how you want a compound expression to be evaluated. For example

a = x + y / 100; //ambiguous!
a = (x + y) / 100; //unambiguous, recommended

If you don't explicitly indicate the order in which you want the operations in a compound expression to be performed, the order is determined by the precedence assigned to the operators. In descending order of importance they are:

Incrementing & Decrementing.
Because adding or subracting one from an integer value is such a common operation, there are shorthand ways to write a statement like this : 

int x = 10;
x = x + 1;   //x = 11
int y = ++x; //y and x = 12
int z = x++; //z = 12, x = 13 
--x;         //decrementing x

There is a difference between the prefix and postfix use of the increment and decrement operators, notice that y is assigned the incremented value of x, while z is assigned the value of x before the increment is performed.
Java also provides the following shorthand notation, that combines the common operations with assignment:

x = x + y;
x += y; //adds y to x
x -= 2; //subtracts 2 from x
x *= 3; //multiplies x by 3
x /= 2; //divides x by 2

Other shorthand operators include, &=, |= ,^=, %=, <<=, >>= .

Note: The bitwise operators only make sense when you use them with the integer datatypes. Do not try to AND, OR or bit shift a floating point variable!

The ternary operator. Is a shorthand (and ugly) way of writing an either / or expression. A conditional expression that evaluates to either true or false must precede the question mark. If that expression is true at runtime, then the value or expression before the colon is evaluated, otherwise the value or expression after the colon is the result. Here is an example, taken from the java.lang.Math class.

/** Returns the absolute value of a double value.
* If the argument is not negative, the argument is returned.
* If the argument is negative, the negation of the argument is returned.
* @param a the double value.
* @return the absolute value of the argument.
*/
public static double abs(double a) {
  return (a <= 0.0D) ? 0.0D - a : a;
}

The bit shift operators >> , >>> and << can be used to shift the bits in an integer datatype left or right a number of places. In combination with the bitwise & (AND) and | (OR) operators discussed below they permit the testing and manipulation of bits in a byte, short, int or long variable.

  • The >> operator, maintains the sign of the integer value while the bit pattern is shifted to the right the specified number of places. The signed shift fills from the left with the sign bit (0 or 1).
  • The >>> operator, does not maintain the sign bit, while shifting to the right. The unsigned shift zero-fills from the left.
  • The << left shift operator can result in 'loss' of the sign bit, and should be used with caution.

Gotcha! Unexpected results can occur when you use these operators with the byte or short datatypes, because the Java runtime promotes them to an int (preserving the sign) to do the shifting, and then truncates the result.

byte b = -1; //signed byte -1 (0xFF) is really (integer 0xFFFFFFFF).
b >>>= 24; //unsigned right shift has no effect! b is still -1 (0xFF)
b >>>= 25; //unsigned right shift has an effect! b is now +127 (0x7F)
 b <<= 4; //signed left shift b is -16 xF0

The bitwise OR operator.
If either bit in the arguments is ‘on’ (1) the corresponding bit in the result will be set on. OR’ing with all bits set ‘off’ (0) has no effect.

byte b = 0;
//set top 4 bits on
b |= 0xF0

The bitwise AND operator.
Turns bits on (1) in the result, only if both bits in the arguments are on. ANDing with all bits set ‘on’ has no effect. This can be used to ‘set’ bits in an integer variable to zero.

byte b = 0xDF;
//top 4 bits unchanged set last 4 bits to zero
b &= 0xF0

Here's an example of how you can use the bitwise operators to set and test bit flags. As a general rule it is better to make use of boolean (true / false) flags, but you may come across code like this : 

static final int VISIBLE = 1;  // binary 0000 0001
static final int DRAGGING = 2; // binary 0000 0010
static final int DRAWING = 4;  // binary 0000 0100
static final int FILLED = 8;   // binary 0000 1000
 int flags = 0; // the bit flags variable

Now to set the "visible" flag when a Shape becomes visible you would use an OR statement:

flags = flags | VISIBLE;

To test a Shape for visibility, you could then write this AND test:

if ((flags & VISIBLE) == VISIBLE) { ... }

The java.math package provides 2 interesting classes.

  • A BigInteger object, can be used to represent an arbitrary precision integer.
  • A BigDecimal should be used when a double does not provide the necessary precision or when more control over rounding is required. It has methods you can use to control the rounding policy, the scale (decimal places) of the decimal value and methods for basic arithmetic and comparison.

BigDecimals and BigIntegers are immutable, which means that once created, they cannot be modified.

The java.text package provides the abstract NumberFormat class, which is the base class for formatting and parsing numbers. It provides a variety of static 'factory methods' that will return a concrete implementation of NumberFormat (actually objects of the subclass DecimalFormat).
A factory method, is a term given to a method that follows the factory design pattern. In general, a factory implementation is useful when you need one object to control the creation of and/or access to other objects. See J2EE design patterns.
Use the getInstance() method or getNumberInstance() to get the normal number format for your country (the default Locale). Use getCurrencyInstance() to get the currency number format. And use getPercentInstance() to get a format for displaying percentages. With this format, a fraction like 0.53 is displayed as 53%.

double E = 2.7182818284590452354;
NumberFormat nf = NumberFormat.getNumberInstance(); //get a number format for default Locale
nf.setMaximumFractionDigits(3); //display 3 decimal places
System.out.println(nf.format(E)); //build a formatted String from the double

A NumberFormat will also parse a String representation of a number, into one of the numeric wrapper classes. However, if the String you pass it, does not represent a valid number value, the parse() method will abruptly terminate, and throw back an object of the ParseException class. We'll talk about Exceptions and how to handle them on a following page.
If the parse() method is successful, it returns an object variable of the abstract java.lang.Number class. This class is a parent to all the other numeric wrapper classes (Integer, Double ..etc), and will reference an instance of one of those classes (an example of polymorphism at work). The Number class has several methods that allow you to extract the numeric value as a fundamental datatype.

String str = "1,201.75"; //str is a sequence of characters
NumberFormat nf = NumberFormat.getNumberInstance();
Number nbr = nf.parse(str); //may throw a ParseException!
double d = nbr.doubleValue(); //d contains the numeric value
