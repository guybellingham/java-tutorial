Java datatypes.

 arithmetic

The Java dataypes are not classes. For performance reasons Java does not construct objects at runtime to contain and handle the fundamental types of data. Instead Java provides the keywords byte, short, int, long, float, double, char, and boolean for you to use to reserve areas of memory to store data values in various formats.

  • Because computer memory and processors are based on a binary (on/off) system, all the data formats are based upon a pattern of bits (binary ones and zeros).
  • Because long sequences of ones and zeros are hard to read, binary numbers (base 2) are often written in a condensed form called hexadecimal (base 16). Where digits from 0 through 9 and letters A through F are used to represent binary numbers 0000 (hex zero) through 1001 (hex 9) and 1010 (hex A) through 1111 (hex F) respectively.
  • A sequence of 8 bits, makes up a byte. A byte can represent numbers from 0 to 255. A sequence of 4 bytes (32 bits) makes up what is known as a 'word' of storage.
  • The Java integer datatypes (byte, short, int and long), are all signed. This means the value is stored in a binary 'twos complement' format, where the most significant bit is reserved for the sign. If the sign bit is 0 the number is positive, and if the sign bit is 1 the number is negative. There is no unsigned keyword in Java.

The java.lang package provides 'wrapper' classes for each of the dataypes. There are some occasions when you want to manipulate an integer or decimal or boolean value as an object. For example, you may want to store a collection of numeric values in a java.util.Vector or one of the other collection classes in the java.util package. Javas collections are defined in terms of java.lang.Object (they can store and retrieve all objects). As shown below, the wrapper classes contain methods to convert between data types and to and from String objects.

Java integer datatypes come in four precisions, byte (8-bit), short (16-bit), int (32-bit) and long (64 bits). By default integer instance variables are initialized to zero.

To interpret a negative twos complement binary number, flip all the bits and add 1 to get the corresponding positive value.

Gotcha! Java will generate an ArithmeticException at runtime if your code attempts to divide an integer value by zero. Java integer arithmetic does not generate values for infinity or imaginary numbers.

Literal integer values can be written as decimal digits 0 thru 9 with a sign. By default the int datatype is used for integer literals. If you are dealing with integer values over 2 billion you should make use of the long datatype which reserves more memory for very large whole number values. If you specify a very large integer literal value, you must suffix it with the letter L, to let the Java compiler know to reserve a long (64-bit) storage area for the value, instead of the default integer (32-bit) storage area.

int x = +2147483647; //is the largest positive integer
int y = -2147482648; //is the smallest integer value
long bigNothing = 65536 * 65536; //integer arithmetic overflows to zero!
long bigNumber = 65536L * 65536; //long arithmetic evaluates to 4294967296

You might be scratching your head at those last 2 lines, the asterisk is Javas way of saying multiply two values.

  • So why does bigNothing end up with zero ? Well 65536 multiplied by itself, is too big to fit into a 32-bit integer. The result overflows and Java ignores that fact and just truncates the most significant bits leaving an answer of zero!
  • Why does bigNumber get the correct value assigned ? Well in this case, Java sees that one of the values is marked as a long value, and it widens the other value to a long also. The arithmetic is done using 64-bits (instead of the default 32-bits) and the result does not overflow. We'll talk more about how Java casts and converts datatypes when doing arithmetic on the following page.

Danger! Java arithmetic does not generate Exceptions when a numeric value overflows or underflows its storage. Apparently this is for robustness, but it is a contentious issue, because the runtime silently truncates the result. In Java arithmetic it is important to use a dataype large enough to hold the largest and smallest expected results.

Tip! Try to avoid coding numeric literals (constants) inside your methods, it is better practice to define them as public static final data variables with an initial value like this :
public static final int DAYS_IN_WEEK = 7;
The convention is to use all CAPS for names of constants.

Integer values can also be expressed as hexadecimal (base 16) numbers, with a prefix of 0x and the digits 0 thru 9 and A thru F (representing decimal values 10 thru 15) :

int x = 0x7FFFFFFF; // is the largest hex integer
int y = 0x80000000; // is the smallest hex integer

You can express integers in octal (base 8), by prefixing the value with a zero, and using the digits 0 thru 7 only. Gotcha! If you write leading zeros on a value, Java may interpret it as an octal value.

Java decimal numbers come in two precisions, float (32-bit) and double (64-bit). By default a decimal literal reserves memory for a double precision value. If you want to explicitly reserve the lesser precision of a float datatype, suffix the literal value with the letter F. The binary format Java uses for floating point numbers and arithmetic is a complex standard called IEEE-754 and is beyond the scope of this guide (see the java.lang.FloatingDecimal class).

Note: Unlike integers, arithmetic on floating point data types does not generate arithmetic errors, because Java can represent values like positive/negative infinity, and imaginary numbers (termed NaN 'Not A Number').
You should use the static class methods isInfinite() and isNaN() in the wrapper classes java.lang.Float and java.lang.Double to test for infinite or imaginary results, don't try to use the equality expression == on NaN values (it will always return false).

Literal decimal values have a whole number part, followed by a decimal point and a fraction, and an optional exponent (E plus/minus a power of ten). By default floating point literals have a datatype of double. If you want a value that can be assigned to a float use a suffix of F or f, for example.

float x = 3.5; //compile error! 3.5 is a double
float x = 3.5F; //OK
float y = 3.402E+38F; //is the largest float
float z = 1.402E-45F; //is the smallest float value
double d = 1.79E+308; //largest double precision value
double e = 4.94E-324; //smallest double

Gotcha! The float dataype has only a limited precision (about 7 significant digits), its use in arithmetic can lead to cumulative rounding errors. Most floating point arithmetic in Java is done using a double (see the java.lang.Math class), and you should follow the example set by the built in classes.

Java characters use a double byte character set (DBCS) 16-bit Unicode format. The Unicode character set supports most writing scripts in use in the world today, including the large sets for Chinese, Japanese, and Korean. This allows programmers to use identifiers in their programs that are written in their native languages. See the Unicode Organization for more details.


Character literals can be represented by a single letter value in single quotes such as 'A' or 'a', an escape sequence in single quotes like '\n' or a Unicode sequence such as ‘\u000A’. The Java compiler ignores 'white space' characters, which are defined as the ASCII space, horizontal tab, and form feed characters, as well as the line terminators (ASCII CR and/or LF).

The Unicode character set,

'\u0030' thru '\u0039' represents the Latin digits 0 - 9
'\u0041' thru '\u005A' are the ASCII Latin lowercase letters
'\u0061' thru '\u007A' are the ASCII uppercase letters

Character escape sequences are needed to represent unprintable and control characters :

'\b' represents a backspace escape code
'\t' is the tab escape code
'\n' is the ASCII LF line feed code
'\r' is the ASCII CR carriage return
'\'' represents a single apostrophe
'\\' is the code for a single back slash
'\"' is a single double quote

Gotcha! You must use single quotes around a character literal, double quotes are used for String literals as we'll see on a following page. The default value for a char is ‘\u0000’ which is not the same as a space, or a null.

The Java boolean dataype can have only 2 values, represented by the keywords true or false. The logical opposites true and false are the only values that may be assigned to a data type of boolean. By default a boolean instance variable is initialized to false.

Gotcha! Unlike other programming languages, Java does not use a numeric value of 0 to represent false and non-zero value for true. So you cannot use an integer value in a conditional test like this :

//This is NOT VALID in Java!
java.util.Random randomizer = new java.util.Random(System.currentTimeMillis());
int r = randomizer.nextInt(2);
if (r) { //do something }

The wrapper classes and some example uses for them are listed below :

java.lang.Byte

//use a String literal to construct a Byte object
Byte myByte = new Byte("127");
byte b1 = myByte.byteValue();
java.lang.Short

Short myShort = new Short("32767");
short s1 = myShort.shortValue();
java.lang.Integer

Integer myInteger = new Integer("2147483647");
int i = myInteger.intValue();
java.lang.Long

//parse a really big base 16 number String
long l = Long.parseLong("7FFFFFFFF", 16);

java.lang.Float

//parse a floating point value in a String
Float myFloat = Float.valueOf("3.402E+38");
java.lang.Double

double d = Double.parseDouble("1.79E+308");
boolean b = Double.isInfinite(d); //returns false
java.lang.Character

char c = 'y';
c = Character.toUpperCase(c);
boolean b = Character.isDigit(c); //returns false
java.lang.Boolean

Boolean myBoolean = new Boolean("true");
boolean b = myBoolean.booleanValue(); //returns true
